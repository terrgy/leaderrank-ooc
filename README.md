# LeaderRank на графах, не помещающихся в память

Программа считает централизованность LeaderRank для **ориентированного невзвешенного**
графа. Главное ограничение задачи — граф не помещается в оперативную память, поэтому рёбра
лежат на диске и читаются потоком, а в памяти остаются только массивы размера O(N) (по числу
вершин). Вычисление идёт в несколько потоков и использует все ядра одного узла. Результат
детерминированный и не зависит от числа потоков.

- **Вход:** CSV с заголовком `from,to`. Каждая строка — ребро `from -> to`, идентификаторы вершин типа int32.
- **Выход:** CSV с заголовком `vertex,rank`.

## Требования

- **JDK 21** (проверялось на OpenJDK 21). Это единственное, что нужно поставить вручную.
- Gradle ставить не надо — в репозитории лежит обёртка (`./gradlew`), она сама скачает нужную версию.
- **Интернет при первой сборке** — Gradle загрузит сам себя и зависимости (Apache Commons CSV, fastutil, picocli, JUnit). Дальше всё работает офлайн.
- Сборка, запуск и тесты работают на **любой ОС** с JDK 21 (Linux, macOS, Windows).
- Скрипты из `scripts/` — **только Linux**. Демонстрация лимита памяти опирается на cgroup v2 и `systemd-run`, бенчмарк использует bash, `curl`, `gunzip`, `awk`. На Windows и macOS они не запустятся. Строка `peak RSS` в выводе тоже только для Linux (читается из `/proc`), на других системах она просто не печатается.

Проверить версию Java:

```bash
java -version    # должно быть 21.x
```

## Сборка

```bash
./gradlew build
```

Эта команда компилирует проект и прогоняет все тесты. На Windows вместо `./gradlew` используйте `gradlew.bat`.

Дальше удобно собрать готовый дистрибутив с лаунчером:

```bash
./gradlew installDist
```

После этого появится исполняемый файл `build/install/leaderrank/bin/leaderrank`. Все примеры ниже используют его. Чтобы не писать длинный путь, можно завести переменную:

```bash
LR=build/install/leaderrank/bin/leaderrank
```

## Запуск

Посчитать ранги для маленького примера из репозитория:

```bash
$LR rank examples/small.csv
```

Записать результат в файл:

```bash
$LR rank examples/small.csv leader_rank.csv
```

По умолчанию берётся потоковое out-of-core представление и все ядра процессора.

Сгенерировать тестовый граф со степенным распределением (в нём появляются гипер-узлы):

```bash
$LR generate rmat.csv --scale=20 --edges=16000000 --seed=42
$LR rank rmat.csv out.csv
```

Проверить корректность — сравнить быстрый движок с прямой эталонной реализацией на одном графе:

```bash
$LR verify examples/small.csv
```

В конце печатается `max abs difference`. На маленьких графах это порядка `1e-15`, то есть совпадение с точностью до ошибки округления. Эталон требует O(n^2) памяти, поэтому `verify` рассчитан только на небольшие графы.

Справка по любой команде:

```bash
$LR --help
$LR rank --help
```

## Полезные опции команды `rank`

- `--threads=P` — число рабочих потоков (по умолчанию все ядра).
- `--engine=parallel|common|dense` — движок. `parallel` по умолчанию, `common` — однопоточный, `dense` — эталонный O(n^2) (только для маленьких графов).
- `--graph=out-of-core|in-memory` — представление графа. `out-of-core` по умолчанию. `in-memory` намеренно держит все рёбра в памяти и нужен для демонстрации (см. ниже).
- `--max-iterations=N` — ограничение числа итераций степенного метода (по умолчанию 100).
- `--print-top=N` — вывести в консоль только N лидеров.
- `--repeat=N` — прогнать вычисление N раз для прогрева JIT, в отчёт идёт последний прогон.

## Ограничение памяти

Размер памяти под графовые структуры программа определяет сама из размера кучи JVM. Достаточно ограничить кучу через `-Xmx`, и движок подстроит под неё размер дисковых корзин.

Куча задаётся переменной `JAVA_OPTS` для лаунчера:

```bash
JAVA_OPTS="-Xmx128m" $LR rank rmat.csv out.csv
```

Пик потребления памяти печатается в строке `peak RSS`. `-Xmx` стоит брать заметно меньше реального предела процесса — около 60-80 МБ уходит на саму JVM поверх кучи.

## Тесты

```bash
./gradlew test
```

Один из тестов проверяет лимит памяти через cgroup. Если запуск не на Linux или `systemd-run --user` не может выставить предел, этот тест автоматически пропускается, остальные проходят как обычно.

## Бенчмарк (только Linux)

```bash
scripts/benchmark.sh           # масштабирование по потокам + реальные графы SNAP
scripts/benchmark.sh quick     # быстрая проверка на маленьком графе
scripts/benchmark.sh scaling   # ускорение при росте числа потоков
scripts/benchmark.sh memory    # зависимость пика RSS от -Xmx
scripts/benchmark.sh snap      # графы web-Google и soc-LiveJournal1
```

Режимы `snap` и `all` скачивают графы с `snap.stanford.edu`, поэтому нужен интернет и около 1 ГБ места под `bench/data/`. Результаты складываются в `bench/results/*.csv`. Параметры можно менять переменными окружения, полный список — в `scripts/benchmark.sh --help`.

## Демонстрация лимита памяти (только Linux, нужен systemd + cgroup v2)

```bash
scripts/memory_cap_demo.sh
```

Скрипт запускает один и тот же алгоритм под жёстким лимитом памяти ядра. Потоковый out-of-core движок укладывается в предел, а наивное представление, которое держит все рёбра в памяти, падает. Это наглядно показывает, ради чего нужен out-of-core подход.
